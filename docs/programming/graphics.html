Graphics and Camera Sensors

Isaac Gym exposes APIs to control visual aspects of the scene programattically. Additionally, Isaac Gym exposes API to manage views from many cameras and to treat these cameras as sensors on the robot. The following sections describe camera properties, camera sensors, visual property modification, and other topics related to graphics and camera sensors.(Isaac Gym 提供了 API 来以编程方式控制场景的视觉方面。此外，Isaac Gym 还提供了 API 来管理多个相机的视图，并将这些相机视为机器人的传感器。以下各节将介绍相机属性、相机传感器、视觉属性修改以及其他与图形和相机传感器相关的主题。)

Camera Properties

All cameras in Isaac Gym, including the viewer’s camera, can be created with parameters passed in a GymCameraProperties structure. The camera properties structure contains properties such as field of view, resolution, near and far clipping distances, and a few other options. When creating the viewer in the normal way:(Isaac Gym 中的所有相机（包括查看器的相机）都可以使用通过 GymCameraProperties 结构传递的参数来创建。相机属性结构包含视场角、分辨率、近平面和远平面剪裁距离以及其他一些选项。当以常规方式创建查看器时：)

viewer = gym.create_viewer(sim, gymapi.CameraProperties())

A camera properties structure with default values is constructed and used for the viewer camera. The properties can be customized as in the following code:(将构造一个具有默认值的相机属性结构并将其用于查看器相机。可以像以下代码那样自定义这些属性：)

camera_props = gymapi.CameraProperties()
camera_props.horizontal_fov = 75.0
camera_props.width = 1920
camera_props.height = 1080
viewer = gym.create_viewer(sim, camera_props)
Camera Properties:(相机属性：)

  • width - image width in pixels(图像宽度（像素）)

  • height - image height in pixels(图像高度（像素）)

  • horizontal_fov - horizontal field of view in radians. The vertical field of view will be height/width * horizontal_fov(水平视场角（弧度）。垂直视场角为 height/width * horizontal_fov)

  • near_plane - distance from the camera point to the near clipping plane in world units (meters)(相机点到近剪裁平面的距离（世界单位，米）)

  • far_plane - distance from the camera point to the far clipping plane in world units (meters)(相机点到远剪裁平面的距离（世界单位，米）)

  • supersampling_horizontal - integer multiplier for supersampling. Setting to a value larger than 1 will result in rendering an image of width*supersampling_horizontal pixels wide and resampling to output resolution(超采样的整数倍率。设置为大于1的值将渲染宽度为 width*supersampling_horizontal 像素的图像，然后重采样到输出分辨率)

  • supersampling_vertical - integer multiplier for supersampling. Setting to a value larger than 1 will result in rendering an image of height*supersampling_vertical pixels tall and resampling to output resolution(超采样的整数倍率。设置为大于1的值将渲染高度为 height*supersampling_vertical 像素的图像，然后重采样到输出分辨率)

  • use_collision_geometry - if set to True, camera will render collision geometry(如果设置为 True，相机将渲染碰撞几何体)

  • enable_tensors - if set to True, tensor access will be enabled for this camera. See GPU Tensor Access section below for more.(如果设置为 True，将为此相机启用张量访问。更多信息请参见下面的 GPU 张量访问 部分。)

Camera Sensors

Camera sensors in Isaac Gym are meant to simulate cameras as they would be found either observing or attached to a robot. A camera sensor can be easily created as follows:(Isaac Gym 中的相机传感器用于模拟在机器人上观察或附加的相机。可以像下面这样轻松创建相机传感器：)

camera_props = gymapi.CameraProperties()
camera_props.width = 128
camera_props.height = 128
camera_handle = gym.create_camera_sensor(env, camera_props)

As with the viewer camera, a customized camera properties can also be passed in to the camera creation. A camera sensor differs from the viewer camera in that it cannot be controled through the viewer and it is connected to a specific environment. This means that when a camera sensor creates an image, it can ‘see’ only those actors in the selected environment.(与查看器相机一样，也可以将自定义的相机属性传递给相机创建。相机传感器与查看器相机的不同之处在于：它不能通过查看器控制，并且它连接到特定的环境。这意味着当相机传感器生成图像时，它只能“看到”所选环境中的那些角色。)

The location of camera sensors can be set in one of three ways. First, a camera sensor may be placed directly using:(相机传感器的位置可以通过三种方式之一来设置。第一种，可以直接使用以下方式放置相机传感器：)

gym.set_camera_location(camera_handle, env, gymapi.Vec3(x,y,z), gymapi.Vec3(tx,ty,yz))

where (x,y,z) are the coordinates of the camera in the environment’s local coordinate frame and (tx, ty,tz) are the coordinates of the point the camera is looking at, again in the environments local coordinates.(其中 (x,y,z) 是相机在环境局部坐标系中的坐标，(tx,ty,tz) 是相机所观察点的坐标（同样在环境局部坐标系中）。)

A second method to place the camera is to directly specify a GymTransform which specifies the position and pose of the camera in environemnt local coordinates, such as:(第二种放置相机的方法是直接指定一个 GymTransform，它指定了相机在环境局部坐标系中的位置和姿态，例如：)

transform = gymapi.Transform()
transform.p = (x,y,z)
transform.r = gymapi.Quat.from_axis_angle(gymapi.Vec3(0,1,0), np.radians(45.0))
gym.set_camera_transform(camera_handle, env, transform)

Finally, a camera can be directly attached to a rigid body. This is useful to create cameras which track a body as it moves or to simulate the camera being directly attached to a part of an actor. Cameras can be attached to a rigid body as follows:(最后，相机可以直接附加到刚体上。这对于创建跟踪移动刚体的相机或模拟相机直接附加到角色某部分非常有用。可以像下面这样将相机附加到刚体上：)

local_transform = gymapi.Transform()
local_transform.p = (x,y,z)
local_transform.r = gymapi.Quat.from_axis_angle(gymapi.Vec3(0,1,0), np.radians(45.0))
gym.attach_camera_to_body(camera_handle, env, body_handle, local_transform, gymapi.FOLLOW_TRANSFORM)

The last argument determines the attachment behavior:(最后一个参数决定了附加行为：)

  • gymapi.FOLLOW_POSITION means that the camera will maintain a fixed offset (local_transform.p) from the rigid body, but will not rotate with the body.(gymapi.FOLLOW_POSITION 表示相机将保持与刚体的固定偏移 (local_transform.p)，但不会随刚体旋转。)

  • gymapi.FOLLOW_TRANSFORM means that the camera will maintain a fixed offset and also rotate with the body.(gymapi.FOLLOW_TRANSFORM 表示相机将保持固定偏移并随刚体旋转。)

When attached to a rigid body, the camera’s transform will be updated in step_graphics(sim).(当附加到刚体时，相机的变换将在 step_graphics(sim) 中更新。)

All camera sensors are rendered together in a single API call to enable best performance:(所有相机传感器都在一个 API 调用中一起渲染，以实现最佳性能：)

gym.render_all_camera_sensors(sim)

After render_all_camera_sensors(sim), all outputs from all cameras will be ready for retrieval.(在 render_all_camera_sensors(sim) 之后，所有相机的所有输出将准备好以供检索。)

Camera Image Types

Each camera renders a number of different output images. When accessing images, APIs take an image selector which indicates which of the output images the application would like to retrieve or access.(每个相机渲染多个不同的输出图像。当访问图像时，API 使用一个图像选择器，该选择器指示应用程序希望检索或访问哪个输出图像。)

Camera Sensor Image Types:(相机传感器图像类型：)

  • IMAGE_COLOR - 4x 8 bit unsigned int - RGBA color(RGBA 颜色)

  • IMAGE_DEPTH - float - negative distance from camera to pixel in view direction in world coordinate units (meters)(相机到像素沿视线方向的负距离（世界单位，米）)

  • IMAGE_SEGMENTATION - 32bit unsigned int - ground truth semantic segmentation of each pixel. See(每个像素的真实语义分割。请参见)

  • IMAGE_OPTICAL_FLOW - 2x 16bit signed int - screen space motion vector per pixel, normalized(每个像素的屏幕空间运动向量，归一化)

The depth image in linearized back into world coordinates. This means that the value of the pixel is in the units of the simulation, which is meters. Each pixel of the segmentation image contains the segmentation ID of the body which generated the pixel. See the section Segmentation Image below. The values returned for the optical flow image are 16 bit signed integers. The following code shows how to convert a value from optical flow image to a value in pixel units:(深度图像被线性化回世界坐标。这意味着像素值以仿真单位（米）表示。分割图像的每个像素包含生成该像素的刚体的分割 ID。请参见下面的 分割图像 部分。光流图像返回的值是 16 位有符号整数。以下代码显示了如何将光流图像的值转换为像素单位的值：)

optical_flow_image = gym.get_camera_image(sim, camera_handle, gymapi.IMAGE_OPTICAL_FLOW)
optical_flow_in_pixels = np.zeros(np.shape(optical_flow_image))
# Horizontal (u)
optical_flow_in_pixels[0,0] = image_width*(optical_flow_image[0,0]/2**15)
# Vertical (v)
optical_flow_in_pixels[0,1] = image_height*(optical_flow_image[0,1]/2**15)

Accessing Camera Images

Images rendered with camera sensors can be accessed by calling get_camera_image for the desired image type as follows:(可以通过为所需图像类型调用 get_camera_image 来访问相机传感器渲染的图像，如下所示：)

color_image = gym.get_camera_image(sim, env, camera_handle, gymapi.IMAGE_COLOR)

The third argument selects which image from that camera should be returned. The above code asks for the color (RGBA) image. The image is returned as a numpy array. When the image contains vector outputs, such as IMAGE_COLOR and IMAGE_OPTICAL_FLOW, these vectors are tightly packed in the fastest moving dimension.(第三个参数选择应从该相机返回哪个图像。上面的代码请求彩色 (RGBA) 图像。图像以 numpy 数组的形式返回。当图像包含向量输出（例如 IMAGE_COLOR 和 IMAGE_OPTICAL_FLOW）时，这些向量在变化最快的维度上紧密排列。)

See graphics example.(请参见 图形示例。)

GPU Tensor Access

When using camera sensor outputs for training a model which is already present on the GPU, a key optimization is to prevent copying the image to the CPU in Isaac Gym only to have the learning framework copy it back to the GPU. To enable better performance, Isaac Gym provides a method for direct GPU access to camera outputs without copying back to CPU memory.(当使用相机传感器输出来训练已存在于 GPU 上的模型时，一个关键的优化是防止在 Isaac Gym 中将图像复制到 CPU，然后又让学习框架将其复制回 GPU。为了实现更好的性能，Isaac Gym 提供了一种直接 GPU 访问相机输出的方法，无需复制回 CPU 内存。)

In order to activate tensor access, a camera must be created with the enable_tensors flag set to True in the camera properties structure.:(为了激活张量访问，必须在相机属性结构中将 enable_tensors 标志设置为 True 来创建相机：)

camera_props = gymapi.CameraProperties()
camera_props.enable_tensors = True
cam_handle = gym.create_camera_sensor(env, camera_props)

During the setup phase, the application should retrieve and store the tensor structure for the camera by calling:(在设置阶段，应用程序应通过调用以下方法检索并存储相机的张量结构：)

camera_tensor = gym.get_camera_image_gpu_tensor(sim, env, cam_handle, gymapi.IMAGE_COLOR)

The returned GPU tensor has a gpu device side pointer to the data resident on the GPU as well as information about the data type and tensor shape. Sharing this data with a deep learning framework requires a tensor adapter, like the one provided in the gymtorch module for PyTorch interop:(返回的 GPU 张量包含指向 GPU 上常驻数据的 GPU 设备端指针，以及有关数据类型和张量形状的信息。要与深度学习框架共享此数据，需要一个张量适配器，例如 gymtorch 模块中为 PyTorch 互操作性提供的适配器：)

camera_tensor = gym.get_camera_image_gpu_tensor(sim, env, cam_handle, gymapi.IMAGE_COLOR)
torch_camera_tensor = gymtorch.wrap_tensor(camera_tensor)

Now during the main loop of the simulation, the application must declare when it starts and stops accessing the tensors to prevent memory hazards.:(现在，在仿真的主循环期间，应用程序必须声明何时开始和停止访问张量，以防止内存冲突：)

while True:

    gym.simulate(sim)
    gym.fetch_results(sim, True)
    gym.step_graphics(sim)
    gym.render_all_camera_sensors(sim)
    gym.start_access_image_tensors(sim)
    #
    # User code to digest tensors
    #
    gym.end_access_image_tensors(sim)

The start_access_image_tensors(sim) API informs Isaac Gym that the user code wants to access the image tensor buffers directly. This API will stall until all of the image tensors from the previous call to render_all_camera_sensors(sim) are written. To minimize stalling, this API is written to stall once for all of the camera sensors which were created with the enable_tensors flag set to True.(start_access_image_tensors(sim) API 通知 Isaac Gym 用户代码想要直接访问图像张量缓冲区。此 API 将等待，直到前一次调用 render_all_camera_sensors(sim) 的所有图像张量都写入完成。为了最大限度地减少等待，此 API 被设计为对所有使用 enable_tensors 标志设置为 True 创建的相机传感器只等待一次。)

Isaac Gym will not write to the tensors again until end_access_image_tensors(sim) is called. By convention, access to image tensors should be started and stopped within the same iteration of the main loop. Calling render_all_camera_sensors(sim) between the start and end of tensor access may lead to an application deadlock.(Isaac Gym 将不会再次写入张量，直到调用 end_access_image_tensors(sim)。按照惯例，对图像张量的访问应在主循环的同一迭代内开始和结束。在张量访问开始和结束之间调用 render_all_camera_sensors(sim) 可能导致应用程序死锁。)

See PyTorch interop example and the chapter on the tensor API for more information.(更多信息请参见 PyTorch 互操作性示例 以及关于 张量 API 的章节。)

Visual Properties

Isaac Gym contains a number of APIs designed to allow programmatic modification of visual attributes for techniques like visual domain randomization. This section briefly touches on the routines for modifying visual attributes of a body or scene.(Isaac Gym 包含多个 API，旨在允许通过编程方式修改视觉属性，用于诸如视觉域随机化等技术。本节简要介绍修改刚体或场景视觉属性的例程。)

Colors

The color of any rigid body can be set with the API:(任何刚体的颜色都可以通过以下 API 设置：)

gym.set_rigid_body_color(env, actor_handle, rigid_body_index, gymapi.MESH_VISUAL_AND_COLLISION, gymapi.Vec3(r, g, b))

In the above call, env is the environment containing the actor, actor_handle is a handle to the actor in the environment and rigid_body_index is the index of the rigid body within the actor whose color is to be modified. The 4th argument is a mesh selector and allows the caller to specify whether to set the color of the rigid body’s visual mesh, MESH_VISUAL, the rigid body’s collision mesh, MESH_COLLISION, or both, MESH_VISUAL_AND_COLLISION. The last argument is a color vector of the new color where red, green, and blue are floating point values between 0.0 and 1.0. This sets the color for the entire rigid body. If the mesh has submeshes with different colors, the color of all submeshes is set to the specified color.(在上述调用中，env 是包含角色的环境，actor_handle 是环境中角色的句柄，rigid_body_index 是角色中要修改颜色的刚体索引。第四个参数是网格选择器，允许调用者指定是设置刚体视觉网格的颜色（MESH_VISUAL）、刚体碰撞网格的颜色（MESH_COLLISION），还是同时设置两者（MESH_VISUAL_AND_COLLISION）。最后一个参数是新颜色的颜色向量，其中 red、green 和 blue 是介于 0.0 和 1.0 之间的浮点值。这将为整个刚体设置颜色。如果网格包含具有不同颜色的子网格，则所有子网格的颜色都将设置为指定颜色。)

The current color of a rigid body can also be queried with:(也可以使用以下方法查询刚体的当前颜色：)

current_color = gym.get_rigid_body_color(env, actor_handle, rigid_body_index, gymapi.MESH_VISUAL)

When get_rigid_body_color is called with the mesh selector MESH_VISUAL_AND_COLLISION, the color of the visual mesh is returned. If the mesh contains submeshes, the color of the first submesh is returned.(当使用网格选择器 MESH_VISUAL_AND_COLLISION 调用 get_rigid_body_color 时，将返回视觉网格的颜色。如果网格包含子网格，则返回第一个子网格的颜色。)

Textures

Isaac Gym provides two routines for creating textures from the API, one that reads a texture from a file and one that accepts a numpy array of pixels to be used as a texture. Textures are created from a file using:(Isaac Gym 提供了两个从 API 创建纹理的例程：一个从文件读取纹理，另一个接受用作纹理的 numpy 像素数组。使用以下方法从文件创建纹理：)

texture_handle = gym.create_texture_from_file(sim, texture_filename);

Valid file types are .jpg, .png, and .bmp. Textures also can be created from a buffer using:(有效的文件类型为 .jpg、.png 和 .bmp。也可以使用缓冲区创建纹理：)

texture_handle = gym.create_texture_from_buffer(sim, width, height, pixelArray)

where pixelArray is a numpy array of type uint8_t with size [height, width*4] of packed RGBA values. Alpha values should always be 255 on input.(其中 pixelArray 是一个 numpy 数组，类型为 uint8_t，形状为 [height, width*4]，包含打包的 RGBA 值。Alpha 值在输入时始终应为 255。)

Both methods for creating a texture allocate a texture within the graphics system, copy the pixel data to the GPU, and finally return a handle that can be used in the future to refer to that texture. Since texture creation requires memory allocation and copy to the GPU, it is strongly recommended that all textures needed in a simulation be created during setup and bodies be modified using the handles during simulation.(这两种创建纹理的方法都会在图形系统中分配纹理，将像素数据复制到 GPU，最后返回一个可用于将来引用该纹理的句柄。由于纹理创建需要内存分配并复制到 GPU，强烈建议在设置期间创建仿真所需的所有纹理，并在仿真期间使用句柄修改刚体。)

The texture of a rigid body can be set in much the same way as the color, with the color replaced by a texture handle returned by either create_texture_from_file() or create_texture_from_buffer() as follows:(刚体的纹理设置方式与颜色非常相似，只需将颜色替换为由 create_texture_from_file() 或 create_texture_from_buffer() 返回的纹理句柄即可，如下所示：)

gym.set_rigid_body_texture(env, actor_handle, rigid_body_index, gymapi.MESH_VISUAL_AND_COLLISION, texture_handle)

Texture, like color, can be queried in the same fashion:(纹理与颜色一样，可以通过相同的方式查询：)

texture_handle = gym.get_rigid_body_texture(env, actor_handle, rigid_body_index, gymapi.MESH_VISUAL)

If there is not a texture assigned to the selected rigid body, INVALID_HANDLE is returned. Like color, if the mesh selector is MESH_VISUAL_AND_COLLISION, the texture assigned to the visual asset, if any, is returned.(如果没有纹理分配给选定的刚体，则返回 INVALID_HANDLE。与颜色类似，如果网格选择器是 MESH_VISUAL_AND_COLLISION，则返回分配给视觉资源的纹理（如果有）。)

Texture coordinates are loaded with the mesh when the asset is created. If a mesh does not contain any texture coordinates, they are generated using a spherical projection at the time the asset is loaded.(纹理坐标在创建资源时与网格一起加载。如果网格不包含任何纹理坐标，则在加载资源时使用球形投影生成它们。)

If a texture is no longer needed it can be freed using:(如果不再需要纹理，可以使用以下方法释放它：)

gym.free_texture(sim, texture_handle)

All textures and mesh data are automatically freed from GPU memory when the GymSim is destroyed.(当 GymSim 被销毁时，所有纹理和网格数据都会自动从 GPU 内存中释放。)

Texture & Color Interaction

Texture and color are multipicatively applied by the renderer. This means that the texture sample at a pixel is multiplied by the body color to calculate the resulting pixel color. For normal application of textures, set the color of the rigid body to (1.0, 1.0, 1.0) when setting the texture to guarantee the correct color. However, this feature can be used to increase the space of domain randomization for a small set of textures by tinting the textures or it could be used to tint a body according to the forces it is experiencing while maintaining its texture.(渲染器将纹理和颜色相乘。这意味着像素处的纹理样本乘以刚体颜色以计算最终的像素颜色。对于正常的纹理应用，在设置纹理时将刚体颜色设置为 (1.0, 1.0, 1.0) 以保证正确的颜色。然而，此功能可用于通过对纹理进行着色来增加少量纹理的域随机化空间，或者可用于根据刚体所受的力对其着色，同时保持其纹理。)

Reset

At any time the color and texture for an entire actor can be reset to the original value loaded from the asset file by calling:(任何时候，可以通过调用以下方法将整个角色的颜色和纹理重置为从资源文件加载的原始值：)

gym.reset_actor_materials(env, actor, gymapi.MESH_VISUAL_AND_COLLISION)

This will reset the materials for all bodies in the actor of the specified type: visual, collision, or both.(这将重置角色中所有刚体的指定类型的材质：视觉、碰撞或两者。)

Segmentation ID

In order to differentiate different parts of the robot in the semantic segmentation ground truth image (IMAGE_SEGMENTATION), it is necessary to assign differing segmentation ID values to differing bodies. The segmentation ID of a body is set much like color and texture:(为了在语义分割真实图像 (IMAGE_SEGMENTATION) 中区分机器人的不同部分，有必要为不同的刚体分配不同的分割 ID 值。刚体的分割 ID 的设置方式与颜色和纹理非常相似：)

gym.set_rigid_body_segmentation_id(env, actor_handle, rigid_body_index, segmentation_id)

When IMAGE_SEGMENTATION is retrieved from a camera, any pixel which originated from the indicated body will have a value of segmentation_id, which is a 32-bit unsigned integer.(当从相机检索 IMAGE_SEGMENTATION 时，源自指定刚体的任何像素的值将为 segmentation_id，这是一个 32 位无符号整数。)

Lights

Isaac Gym’s rendering has a limited set of lights that can be controlled programatically with the API:(Isaac Gym 的渲染具有一组有限的光源，可以通过 API 以编程方式进行控制：)

gym.set_light_parameters(sim, light_index, intensity, ambient, direction)

light_index is the index of the light, only values 0 throuhg 3 are valid . Intensity is a Vec3 of the relative RGB values for the light intensity, where 0 is off and 1.0 is max intensity. There is no physical correspondence for the intensity values. ambient is a Vec3 of RGB values for the ambient lighting in the same fashion as the classical phong model. Finally, direction is a Vec3 which gives the direction of the light. All lights are directional lights only.(light_index 是光源的索引，仅值 0 到 3 有效。intensity 是一个 Vec3，表示光源强度的相对 RGB 值，其中 0 表示关闭，1.0 表示最大强度。这些强度值没有对应的物理含义。ambient 是一个 Vec3，表示环境光的 RGB 值，方式与经典 Phong 模型相同。最后，direction 是一个 Vec3，表示光的方向。所有光源都只是方向光。)

The lighting model ise the only part of Isaac Gym’s graphics system which is shared globally. Calling set_light_parameters affects the RGB images of all camera sensors in all environments and the viewer universally.(光照模型是 Isaac Gym 图形系统中唯一全局共享的部分。调用 set_light_parameters 会影响所有环境中所有相机传感器的 RGB 图像以及查看器。)

Lines API

The lines API is a simple API for drawing lines on top of the rendered image in the viewer for debugging or visualization purposes. Lines can be used to visualize contacts, applied forces, or any other vector quantity reasonably easily. The lines API is very rudimentary, with one API to add lines to an accumulator of lines, and another API to flush all lines. The API to add lines is as follows:(Lines API 是一个简单的 API，用于在查看器的渲染图像上绘制线条，用于调试或可视化目的。线条可以用于可视化接触点、施加的力或任何其他向量量，相当容易。Lines API 非常基础，有一个 API 用于将线条添加到线条累加器中，另一个 API 用于清除所有线条。添加线条的 API 如下：)

gym.add_lines(viewer, env, num_lines, vertices, colors)

In this call, vertices is an array of size [num_lines * 2 , 3] where element [2 * i, :] specifies a 3D coordinate in the local frame of env for the start and element 2*i+1 specifies the coordinate of the end of line i. colors is an array of size [num_lines, 3] where every index [i,:] is an RGB color for line i.(在此调用中，vertices 是一个形状为 [num_lines * 2, 3] 的数组，其中元素 [2 * i, :] 指定线条 i 起点在 env 局部坐标系中的 3D 坐标，元素 2*i+1 指定线条 i 终点的坐标。colors 是一个形状为 [num_lines, 3] 的数组，其中每个索引 [i,:] 是线条 i 的 RGB 颜色。)

All of the lines that have been added for drawing are removed with a call to:(所有已添加用于绘制的线条都通过调用以下方法移除：)

gym.clear_lines(viewer)

A common use for the lines API is to add lines right before drawing the viewer and clearn them immediately afterward:(Lines API 的一个常见用法是在绘制查看器之前添加线条，然后立即清除它们：)

while not gym.query_viewer_has_closed(viewer):
    gym.simulate(sim)
    gym.fetch_results(sim, True)
    gym.step_graphics(sim)

    # generate_lines is a user-written function which creates the desired line vertices
    line_vertices, line_colors, num_lines = generate_lines()

    gym.add_lines(viewer, env, num_lines, line_vertices, line_colors)
    gym.draw_viewer(viewer)

Lines drawn with the Lines API are only visible in the viewer. They are not rendered into the view of camera sensors.(使用 Lines API 绘制的线条仅在查看器中可见。它们不会渲染到相机传感器的视图中。)

Camera Intrinsics

Some advanced uses of Isaac Gym, such as deprojecting depth images to 3D point clouds, requires complete knowledge of the projection terms used to create the output images. To aid in this, Isaac Gym provides access to the projection and view matrix used to render a camera’s view. They are:(Isaac Gym 的一些高级用法，例如将深度图像反投影到 3D 点云，需要完全了解用于创建输出图像的投影参数。为了帮助实现这一点，Isaac Gym 提供了对用于渲染相机视图的投影矩阵和视图矩阵的访问。它们是：)

projection_matrix = np.matrix(gym.get_camera_proj_matrix(sim, env, camera_handle))

view_matrix = np.matrix(gym.get_camera_view_matrix(sim, env, camera_handle))

Both of these functions return a numpy array, which is converted to a numpy matrix in the code samples above.(这两个函数都返回一个 numpy 数组，在上面的代码示例中将其转换为 numpy 矩阵。)

Step Graphics

When rendering is required, transforms and information must be communicated from the physics simulation into the graphics system. In order to support use cases in which graphics and physics are not running at the same update rate, e.g. if graphics is rendering only every Nth step, Isaac Gym allows manual control over this process. This is further important when many camera sensors are used as the renderer must know when it has moved to a new frame in time, but render() calls are not implicitly frame boundaries since there may be many render() calls per step for viewers and camera sensors. The frame boundary update is made explicit with the step_graphics(sim) API.(当需要渲染时，必须将变换和信息从物理仿真传递到图形系统。为了支持图形和物理不以相同更新率运行的情况（例如，如果图形仅每 N 步渲染一次），Isaac Gym 允许手动控制此过程。当使用多个相机传感器时，这一点更为重要，因为渲染器必须知道它何时及时移动到了新帧，但 render() 调用并不隐式地是帧边界，因为对于查看器和相机传感器，每一步可能有多个 render() 调用。帧边界更新通过 step_graphics(sim) API 显式指定。)

In most cases, however, step_graphics() will be called immediately before a call to draw_viewer() and/or render_all_camera_sensors() as follows:(然而，在大多数情况下，step_graphics() 将在调用 draw_viewer() 和/或 render_all_camera_sensors() 之前立即调用，如下所示：)

while not gym.query_viewer_has_closed(viewer):
    gym.simulate(sim)
    gym.fetch_results(sim, True)

    gym.step_graphics(sim)
    gym.render_all_camera_sensors(sim)
    gym.draw_viewer(viewer)
